# 09.4 SQL с нуля: SELECT, WHERE, ORDER BY

Представьте свой первый рабочий день. Вам открыли доступ, вы запустили DBeaver, подключились к тестовой базе — и перед вами пустое поле для запроса и мигающий курсор. С чего начать?

В программировании первую программу традиционно начинают с «Hello, world». В мире баз данных свой набор первых слов. Всё крутится вокруг нескольких операций, которые складываются в аббревиатуру **CRUD**:

- **C — Create** (создать) — команда `INSERT`, добавить новые данные.
- **R — Read** (прочитать) — команда `SELECT`, получить данные.
- **U — Update** (обновить) — команда `UPDATE`, изменить существующие.
- **D — Delete** (удалить) — команда `DELETE`, убрать данные.

Для аналитика из этой четвёрки главная — и часто единственная — буква **R**. Мы **читаем** данные: выбираем, фильтруем, считаем. Создавать, менять и удалять на боевой базе аналитику обычно нельзя, и это правильно (почему — разберём в [[09.9 Как читать чужую схему данных|09.9]]). Поэтому дальше — про `SELECT`, рабочую лошадку аналитика.

> **Для полноты картины — как выглядят остальные три.** Выполнять их на бою вы не будете, но узнавать в чужом коде должны:
> ```sql
> INSERT INTO clients (lastname, status) VALUES ('Казгамбаев', 'active'); -- добавить строку
> UPDATE clients SET status = 'blocked' WHERE id = 5;                     -- изменить существующую
> DELETE FROM clients WHERE id = 5;                                       -- удалить
> ```
> Присмотритесь к `WHERE` в последних двух командах. Забудете его в `UPDATE` или `DELETE` — и изменение применится не к одной строке, а ко **всей** таблице. Один такой промах способен обнулить статус у всех клиентов разом. Именно поэтому прав на изменение аналитику на боевой базе не дают (подробнее — [[09.9 Как читать чужую схему данных|09.9]]).

> **Для тех, кто уже в теме:** удаляющих команд на самом деле три, и на собеседованиях любят спрашивать разницу. `DELETE` убирает строки по условию `WHERE`, и его можно откатить в рамках транзакции. `TRUNCATE` мгновенно очищает **всю** таблицу от строк (структура остаётся), быстрее `DELETE`, но без разбора. `DROP` удаляет саму таблицу целиком — вместе со структурой и данными. Правило памяти: `DELETE` — про строки, `TRUNCATE` — про содержимое, `DROP` — про объект. Не путайте команду `TRUNCATE` с функцией `TRUNC` из [[09.7 Процедуры, функции, сессии и джобы|09.7]] — это разные вещи с похожим именем.

## Первая выборка: SELECT

Пусть у нас есть таблица `clients` с полями: `lastname` (фамилия), `firstname` (имя), `birthdate` (дата рождения) и `status` (статус клиента).

Достанем всё, что в ней есть:

```sql
SELECT * FROM clients;
```

Читается по-человечески: «выбери (`SELECT`) все столбцы (`*`) из (`FROM`) таблицы `clients`». Нажимаем «выполнить» — и получаем список всех клиентов.

Поздравляю, это ваша первая выгрузка.

Звёздочка `*` означает «все столбцы». Для быстрого «глянуть, что в таблице» — удобно. Но как только вы понимаете, что вам нужны только фамилия и статус, лучше перечислить столбцы явно:

```sql
SELECT lastname, status FROM clients;
```

> **Хорошая привычка с самого начала:** на реальных проектах `SELECT *` считается дурным тоном для рабочих запросов. Он тянет все столбцы, включая тяжёлые и ненужные, грузит базу и делает запрос хрупким. Правило простое: для разведки — можно `*`, для дела — перечисляй столбцы.

## Фильтр: WHERE

Вся таблица целиком нужна редко. Обычно вопрос звучит конкретнее: «покажи только активных клиентов». За фильтрацию отвечает `WHERE` (где):

```sql
SELECT * FROM clients WHERE status = 'active';
```

Теперь в выгрузке — только строки, где статус равен `active`. Условие можно усложнять привычными операторами:

```sql
SELECT * FROM clients WHERE status = 'active' AND birthdate > '1990-01-01';
```

Несколько операторов, которые закроют большинство задач:

| Оператор | Значение | Пример |
|---|---|---|
| `=`, `<>` | равно / не равно | `status = 'active'` |
| `>`, `<`, `>=`, `<=` | больше / меньше | `birthdate > '1990-01-01'` |
| `AND`, `OR` | и / или | `status = 'active' AND ...` |
| `IN` | входит в список | `status IN ('active', 'vip')` |
| `BETWEEN` | в диапазоне | `birthdate BETWEEN '1990-01-01' AND '2000-01-01'` |
| `LIKE` | по шаблону текста | `lastname LIKE 'Ив%'` (начинается на «Ив») |
| `IS NULL` | значение отсутствует | `status IS NULL` |

Обратите внимание на последнюю строку: проверка на «пусто» пишется не `= NULL`, а `IS NULL`. Это та самая ловушка `NULL`, о которой шла речь раньше, — запомните её сразу, она экономит часы недоумения.

## Сортировка: ORDER BY

Данные из базы приходят в непредсказуемом порядке — база не обязана отдавать их «по алфавиту». Чтобы упорядочить результат, в конец запроса добавляют `ORDER BY`:

```sql
SELECT * FROM clients WHERE status = 'active' ORDER BY lastname;
```

На выходе — активные клиенты, отсортированные по фамилии от А до Я. По умолчанию сортировка идёт по возрастанию (`ASC`). Нужно наоборот, от Я до А или от большего к меньшему — добавляем `DESC`:

```sql
SELECT * FROM clients ORDER BY birthdate DESC;
```

— сначала самые молодые.

## Убрать дубликаты и ограничить вывод: DISTINCT и LIMIT

Ещё два коротких, но постоянно нужных инструмента.

**`DISTINCT`** убирает повторяющиеся строки из результата — оставляет только уникальные значения. Вопрос «а какие вообще статусы бывают у клиентов» решается одной строкой:

```sql
SELECT DISTINCT status FROM clients;
```

Вместо миллиона строк — короткий список: `active`, `blocked`, `vip`. Это, между прочим, лучший способ разведать незнакомый столбец: не гадать, что в нём лежит, а прямо спросить у базы. `DISTINCT` работает и по нескольким столбцам сразу — тогда уникальной считается их комбинация (например, `SELECT DISTINCT city, status` покажет все встречающиеся пары «город + статус»).

**`LIMIT`** ограничивает количество строк в ответе. Когда таблица огромная, а нужно просто заглянуть внутрь, не тяните всё:

```sql
SELECT * FROM clients LIMIT 10;
```

— первые десять строк. Синтаксис здесь у диалектов разный: в PostgreSQL и MySQL это `LIMIT 10`, в Oracle исторически `ROWNUM <= 10` или `FETCH FIRST 10 ROWS ONLY`, в MS SQL — `SELECT TOP 10`. Идея у всех одна. Заведите привычку начинать знакомство с любой таблицей именно с ограниченной выборки — это бережёт и базу, и ваше время.

## Собираем всё вместе

Разберём готовый запрос как фразу — именно так его стоит читать, по частям:

```sql
SELECT lastname, firstname          -- какие столбцы показать
FROM clients                        -- из какой таблицы
WHERE status = 'active'             -- какие строки оставить
ORDER BY lastname ASC;              -- как отсортировать
```

«Покажи фамилию и имя из таблицы клиентов, только активных, по алфавиту». Четыре строки — и вы ответили на реальный рабочий вопрос без единого обращения к разработчику.

Хороший способ закрепить это — не читать, а делать. Алан Бьюли в книге «Изучаем SQL» советует ровно это: развернуть учебную базу и прогонять каждый пример руками, меняя условия и глядя, что изменилось в результате. Онлайн-тренажёры (вроде SQL Academy) дают то же самое без установки: пара десятков решённых задач — и `SELECT` перестаёт пугать. Для уровня, который спрашивают на большинстве собеседований, этого уже почти достаточно.

---

Итог. `SELECT` выбирает столбцы, `FROM` указывает таблицу, `WHERE` фильтрует строки, `ORDER BY` сортирует, а `DISTINCT` и `LIMIT` помогают быстро осмотреться в незнакомой таблице. Это базовый набор, из которого собран почти любой запрос аналитика. Освойте его руками на реальных данных — и вы уже полезны команде. Дальше усложняем: научимся собирать данные не из одной таблицы, а из нескольких — через `JOIN`.
