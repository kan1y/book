# 09.5 JOIN: соединяем таблицы

В прошлой подглаве мы доставали данные из одной таблицы. Но помните главную идею реляционной базы: данные специально разложены по разным таблицам, чтобы не дублировались. Клиенты — в `clients`, их заказы — в `orders`. И почти сразу возникает вопрос, на который одна таблица ответить не может: «покажи фамилию клиента и сумму его заказа». Фамилия — в одной таблице, сумма — в другой. Их нужно **соединить**. Это делает `JOIN`.

Умение джойнить таблицы — тот самый рубеж, который на многих собеседованиях называют «средним уровнем SQL». Разберёмся, и рубеж будет взят.

---

## Как работает соединение

Есть две таблицы. `clients`:

| id | lastname |
|---|---|
| 1 | Казгамбаев |
| 2 | Валяев |
| 3 | Искаков |
| 4 | Айтжанов |

И `orders` (в ней `client_id` — внешний ключ, ссылка на `clients.id`):

| id | client_id | sum |
|---|---|---|
| 501 | 1 | 1200 |
| 502 | 1 | 340 |
| 503 | 2 | 900 |

Задача: получить фамилию клиента рядом с суммой заказа. Соединяем таблицы по условию «`client_id` в заказах равен `id` в клиентах»:

```sql
SELECT clients.lastname, orders.sum
FROM clients
JOIN orders ON clients.id = orders.client_id;
```

Ключевое место — `ON clients.id = orders.client_id`. Это правило соединения: база берёт каждую строку заказа и подставляет к ней клиента с совпадающим `id`. Результат:

| lastname | sum |
|---|---|
| Казгамбаев | 1200 |
| Казгамбаев | 340 |
| Валяев | 900 |

Обратите внимание: Казгамбаев встречается дважды — у него два заказа. А Искаков (`id = 3`) и Айтжанов (`id = 4`) не попали вовсе — у них заказов нет. И вот здесь начинается самое важное: **что делать с теми, у кого пары нет?** Ответ зависит от вида JOIN.

---

## Четыре вида JOIN

**INNER JOIN** (или просто `JOIN`) — «только совпадения». Возвращает строки, для которых нашлась пара в обеих таблицах. Клиент без заказов и заказ без клиента отсекаются. В примере выше Искаков и Айтжанов выпали именно поэтому.

**LEFT JOIN** — «все из левой таблицы плюс совпадения из правой». Берём всех клиентов, даже без заказов; там, где заказа нет, в его столбцах будет `NULL` (пусто).

```sql
SELECT clients.lastname, orders.sum
FROM clients
LEFT JOIN orders ON clients.id = orders.client_id;
```

| lastname | sum |
|---|---|
| Казгамбаев | 1200 |
| Казгамбаев | 340 |
| Валяев | 900 |
| Искаков | *NULL* |
| Айтжанов | *NULL* |

Теперь Искаков и Айтжанов на месте — с пустой суммой. Именно `LEFT JOIN` отвечает на вопрос «а есть ли клиенты, которые вообще ничего не заказывали». Такие вопросы аналитику прилетают постоянно.

**RIGHT JOIN** — зеркальное отражение: «все из правой таблицы плюс совпадения из левой». На практике его используют реже — обычно тот же смысл удобнее выразить через `LEFT JOIN`, просто поменяв таблицы местами.

**FULL OUTER JOIN** — «все из обеих таблиц». И клиенты без заказов, и заказы без клиента (если такие затесались) — всё попадёт в результат, недостающие стороны заполнятся `NULL`. Нужен, когда важно не потерять ничего ни с одной стороны.

[ТРЕБУЕТСЯ ИЛЛЮСТРАЦИЯ: четыре вида JOIN как диаграммы Венна (два пересекающихся круга «Клиенты» и «Заказы») с подсветкой области, которую возвращает каждый — INNER, LEFT, RIGHT, FULL]

## Псевдонимы: чтобы не писать длинно

Писать `clients.lastname` и `orders.sum` целиком быстро надоедает. Таблицам дают короткие псевдонимы (алиасы) через пробел или `AS`:

```sql
SELECT c.lastname, o.sum
FROM clients c
JOIN orders o ON c.id = o.client_id;
```

`c` — это `clients`, `o` — `orders`. Тот же запрос, но читается легче. В реальных запросах с тремя-четырьмя таблицами без псевдонимов уже не обойтись.

---

## Как выбрать нужный JOIN

Вопрос, который стоит задавать каждый раз: **«а что делать со строками без пары?»**

- Нужны только те, у кого пара есть (клиенты с заказами) → `INNER JOIN`.
- Нужны все из основной таблицы, даже одиночки (все клиенты, заказы если есть) → `LEFT JOIN`.
- Нельзя потерять ничего ни с одной стороны → `FULL OUTER JOIN`.

Девять из десяти рабочих задач закрываются `INNER` и `LEFT`. С них и начинайте.

> **Для тех, кто уже в теме:** классическая ловушка — `LEFT JOIN` с условием по правой таблице, вынесенным в `WHERE`. Строчка `WHERE o.sum > 500` после `LEFT JOIN` молча превращает его в `INNER`: строки с `NULL` не проходят условие и отсекаются, и клиенты без заказов, которых вы хотели сохранить, исчезают. Если условие относится к «необязательной» стороне — его место в `ON`, а не в `WHERE`. На этой мелочи горят даже крепкие середняки.

---

Запомните главное про JOIN: он не про синтаксис, он про вопрос «что делать с теми, у кого нет пары». `INNER` — оставить только совпадения, `LEFT` — сохранить всех из основной таблицы. Ответьте на этот вопрос — и нужный JOIN выберется сам. Дальше научимся не просто соединять строки, а считать по ним итоги — через группировку.
