# 06.2 Форматы — ТЗ, user story, use case

[ПРИМЕР АВТОРА: нужен случай — например, когда пришёл на проект и команда работает по Agile, а ты привык к полноценному ТЗ — или наоборот]

Формат документа выбирается исходя из методологии, масштаба задачи и того, кто будет это читать. В Agile-команде с двухнедельными спринтами никто не будет читать 80-страничное ТЗ по ГОСТу. В госпроекте с интеграцией через Налоговый комитет — никто не примет неформальную пользовательскую историю.

Разберём три основных формата — когда и для чего.

---

## Техническое задание (ТЗ)

ТЗ — документ старой закалки. Его время — **Waterfall** (каскадная модель разработки), когда весь проект планируется заранее и реализуется последовательно: сначала анализ, потом проектирование, потом разработка, потом тестирование.

Если работаете с государственными структурами — будьте готовы: ТЗ там пишется по ГОСТу 34. Настоящий мазохизм на 100+ страниц с разделами «Общие сведения», «Назначение и цели создания системы», «Характеристика объектов автоматизации» и так далее. Зато формально — всё чисто и в случае споров есть на что сослаться.

В коммерческих проектах ТЗ встречается в упрощённом виде. Хорошее ТЗ содержит:
- **Цель** — зачем вообще делаем
- **Функциональные требования** — что система должна делать
- **Нефункциональные требования** — как должна делать (производительность, безопасность, доступность)
- **Ограничения** — что не делаем и почему
- **Приложения** — схемы, макеты, примеры запросов

ТЗ хорошо подходит для крупных интеграций, когда нужно зафиксировать всё «на берегу» — например, когда подрядчик делает работу по договору и акт будет подписываться именно по документу.

---

## Epic → User Story → Task (Agile-иерархия)

С внедрением Agile-культуры задачи стали дробиться. Большие зоны разработки оцениваются в сторипоинтах или «майках» (S/M/L/XL — как размеры футболок), и появилась трёхуровневая иерархия:

**Epic (Эпик)** — большая целостная задача. Настолько большая, что её нельзя взять в один спринт. Например: «добавить в мобильное приложение раздел Переводы» — задача на месяцы. Эпик описывает верхнеуровневый контекст и разбивается на пользовательские истории. К нему удобно приложить BPMN-диаграмму, которая показывает полный процесс целиком.

[ТРЕБУЕТСЯ ИЛЛЮСТРАЦИЯ: схема иерархии Epic → User Story → Task в виде дерева или пирамиды с примерами]

**User Story (US, пользовательская история)** — кусок эпика, который закрывает конкретную потребность конкретного пользователя. Условно треть или четверть пирога. В US описывается бизнес-логика: что пользователь хочет, зачем и при каких условиях. Подробнее — в следующей подглаве.

**Task (Задача)** — техническая единица, которая уходит разработчику в работу. Конкретно: создать таблицу с описанием полей, написать новый эндпоинт, настроить пуш-уведомления через брокер сообщений. Никакой «бизнес-логики» — только то, что нужно сделать руками.

---

## Use Case (Сценарий использования)

Use case — описание того, **как пользователь взаимодействует с системой** для достижения конкретной цели. Менее жёсткий формат, чем ТЗ, и более детальный, чем user story.

Use case отвечает на вопросы: кто участвует в процессе, какой основной сценарий, что происходит при ошибках и исключениях. Он удобен для стейкхолдеров — они видят процесс глазами пользователя — и для тестировщиков, которые видят, что именно нужно проверять. Часто use case оформляется как подзадача к user story.

---

## Когда что использовать

| Ситуация | Формат |
|---|---|
| Госпроект, интеграция с регулятором | ТЗ по ГОСТу |
| Крупный коммерческий проект с фиксированным объёмом | ТЗ (упрощённое) |
| Agile-команда, продуктовая разработка | Epic + User Story + Task |
| Нужно показать процесс стейкхолдерам и тестировщикам | Use Case |

В реальности форматы часто смешиваются: в Agile-проекте может существовать верхнеуровневое ТЗ для договора и user stories для спринтов. Главное — понимать, кто будет читать документ и что ему нужно из него узнать.

---

**Итог.** ТЗ, user story и use case — не конкуренты, а разные инструменты под разные задачи. Выбор формата зависит от методологии команды, масштаба задачи и аудитории документа. Аналитик должен уметь работать с каждым — и знать, когда что уместно.
