# 08.3 UML для аналитика: три диаграммы, которые реально нужны

Знакомый аналитик вспоминал свой первый проект, где уже была UML-документация: папка в Confluence, схемы на каждый сервис, диаграммы классов и последовательностей. Первые две недели он открывал их, смотрел на прямоугольники со стрелками — и закрывал: понять без контекста было невозможно. Потом ему объяснили: достаточно освоить три диаграммы, остальные — для редких случаев. После этого работать с документацией стало в разы проще. То же самое ждёт большинство аналитиков при первом знакомстве с UML — ощущение чужого языка, который при ближайшем рассмотрении оказывается куда меньше, чем кажется.

**UML** (Unified Modeling Language — унифицированный язык моделирования) — это стандарт, который придумали в 1990-х, чтобы объектно-ориентированные системы можно было описывать графически. Сегодня UML включает 14 видов диаграмм: от диаграммы классов до диаграммы временной шкалы. Если открыть учебник по UML — он легко потянет на шестьсот страниц.

Хорошая новость: аналитику не нужны все 14. На практике активно используются три-четыре, и именно они появляются на проектах снова и снова.

## Use Case Diagram — кто что может делать в системе

**Диаграмма вариантов использования** (Use Case Diagram) показывает, кто взаимодействует с системой и что может делать. Это самая «высокоуровневая» из UML-диаграмм — здесь нет деталей реализации, только функциональный периметр.

Три ключевых элемента:
- **Актор** (actor) — кто взаимодействует с системой. Рисуется как фигурка человека. Это может быть пользователь, внешняя система или устройство.
- **Вариант использования** (use case) — что можно сделать с системой. Рисуется как овал с названием: «Оформить заявку», «Просмотреть отчёт», «Пополнить баланс».
- **Связь** (association) — линия между актором и вариантом использования.

Когда использовать: когда нужно показать стейкхолдеру, что система умеет в целом. Хорошо работает на этапе формирования требований — перед тем как уходить в детали. Заказчик смотрит на Use Case и говорит: «Вот это нам не нужно, а вот это обязательно».

## Sequence Diagram — как компоненты разговаривают между собой

**Диаграмма последовательности** (Sequence Diagram) показывает, как объекты или системы обмениваются сообщениями во времени. Это самая полезная диаграмма для аналитика, работающего с интеграциями.

Как читать: по горизонтали — участники (системы, компоненты, роли). По вертикали — время (сверху вниз). Стрелки между участниками — сообщения, запросы, ответы. Пунктирная стрелка означает ответ, сплошная — запрос.

В одной команде аналитик описывал OAuth-авторизацию через сторонний провайдер текстом — три абзаца с порядком вызовов. Разработчик прочитал, сказал «понял» — и через день задал пять уточняющих вопросов. Аналитик нарисовал Sequence Diagram: семь участников, двенадцать стрелок. Разработчик посмотрел пять минут, задал два вопроса и ушёл реализовывать. Тот же объём информации — другое восприятие.

Типичный сценарий: описать API-взаимодействие между двумя сервисами. Пользователь нажимает «Оплатить» → фронтенд отправляет запрос в бэкенд → бэкенд запрашивает платёжный сервис → получает ответ → возвращает статус пользователю. Такую цепочку можно написать текстом, а можно нарисовать Sequence Diagram — и разработчик за пять секунд поймёт порядок вызовов.

## State Machine Diagram — как меняются состояния объекта

**Диаграмма конечных автоматов** (State Machine Diagram) описывает, как объект переходит из одного состояния в другое под воздействием событий. Особенно полезна для описания жизненного цикла сущностей: заказ («создан» → «оплачен» → «в обработке» → «доставлен» → «закрыт»), заявка, договор, пользователь.

Аналитик часто использует её, когда в требованиях появляется фраза «статус объекта». Если у объекта есть статус — значит, у него есть состояния, и их стоит нарисовать. Часто именно на этом этапе выясняется: а что происходит, если заказ отменили уже после оплаты? А если доставка не состоялась? Диаграмма вскрывает пропущенные сценарии.

## Activity Diagram — поток работ с параллельными ветками

**Диаграмма активностей** (Activity Diagram) внешне похожа на блок-схему и частично пересекается с BPMN. Она показывает поток управления: что за чем происходит, где есть ветки и параллельные активности.

Главное отличие от BPMN: Activity Diagram ближе к коду. Её удобно использовать для описания алгоритмов, логики обработки ошибок, вариантов поведения системы. BPMN лучше работает для бизнес-процессов с участием людей и отделов.

На практике аналитик выбирает между ними по ситуации: BPMN — когда процесс описывается бизнесу, Activity Diagram — когда нужна более формальная модель для разработчика.

> **Для тех, кто уже в теме:** В UML-сообществе существуют расширения Use Case диаграммы: отношения `<<include>>` (обязательно включает другой вариант использования) и `<<extend>>` (опционально расширяет). В учебниках им уделяют много внимания, на практике — меньше. Используйте, только если диаграмма без них перестаёт читаться.

## Инструменты для UML

- **PlantUML** — текстовый формат: описываете диаграмму кодом, получаете картинку. Удобно хранить рядом с кодом в репозитории. Работает в Confluence, VS Code, IntelliJ IDEA.
- **draw.io (diagrams.net)** — есть готовые UML-шаблоны, можно рисовать мышкой. Бесплатный.
- **Lucidchart** — платный, но удобный онлайн-инструмент с хорошей совместной работой.
- **StarUML** — десктопный инструмент для более серьёзного UML-моделирования.

---

UML не нужно учить весь. Освойте Use Case для первого разговора с заказчиком, Sequence для интеграций и State Machine для объектов с жизненным циклом — и вы закроете 80% ситуаций, в которых аналитику нужна диаграмма. Остальное придёт по мере необходимости.
