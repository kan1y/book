# 08.7 Инструменты: где рисовать и как не потерять схему

[ПРИМЕР АВТОРА: нужен случай, когда диаграмма была нарисована, но потом потерялась или устарела — или когда нашли схему, которую никто не обновлял месяцами]

Есть классическая история на проектах: аналитик нарисовал отличную схему, показал команде, все кивнули, схема ушла в папку. Через полгода кто-то находит её, показывает новому разработчику — а там описана старая логика, которую уже трижды переделали. Никто не виноват: схему просто забыли обновить.

Инструмент и место хранения — не мелочь. Если схема живёт «нигде», она быстро устаревает. Если она не открывается у половины команды — не используется. Хороший инструмент — тот, который вписывается в рабочий процесс команды, а не тот, который звучит профессиональнее.

---

## Карта инструментов

**Для совместного рисования на встречах:**
- **Miro** — онлайн-доска, в которой можно работать одновременно. Есть готовые шаблоны для BPMN, Event Storming, UML. Хорошо работает на удалённых воркшопах. Условно-бесплатный.
- **FigJam (Figma)** — аналог Miro, удобен если команда уже работает в Figma.

**Для создания точных диаграмм:**
- **draw.io / diagrams.net** — бесплатный, мощный, есть все нужные элементы. Схемы можно хранить в Google Drive, в репозитории (файл в XML-формате), в Confluence. Хороший выбор по умолчанию для большинства задач.
- **bpmn.io** — специализированный редактор именно для BPMN. Строгий по нотации, экспортирует в XML-формат BPMN 2.0.
- **Lucidchart** — удобный интерфейс, хорошая совместная работа, платный для расширенного функционала.

**Для интеграции с кодом и документацией:**
- **PlantUML** — описываете диаграмму текстом в специальном синтаксисе, инструмент генерирует картинку. Работает прямо в Confluence, VS Code, IntelliJ IDEA. Схемы можно хранить в git-репозитории рядом с кодом.
- **Mermaid** — похожий подход, но ориентирован на Markdown. Поддерживается в GitHub, GitLab, Notion, Obsidian. Если документация пишется в Markdown — удобный вариант без лишних инструментов.

## Diagrams-as-code: когда схема живёт в репозитории

Есть подход, который называют **diagrams-as-code** (диаграммы как код). Суть: схема описывается в текстовом формате и хранится в git-репозитории рядом с кодом или документацией.

Преимущества:
- Схема **версионируется** вместе с кодом. Если логика изменилась — в коммите видно, что поменялось и в коде, и на схеме.
- Схему **легко найти**: она лежит там же, где соответствующий модуль или сервис.
- Изменения **легко сравнивать**: текстовый diff показывает, что именно изменилось в диаграмме.

Пример диаграммы последовательности в синтаксисе Mermaid:

```
sequenceDiagram
    Клиент->>Бэкенд: POST /orders
    Бэкенд->>БД: INSERT order
    БД-->>Бэкенд: OK
    Бэкенд-->>Клиент: 201 Created
```

Это обычный текст, который GitHub и GitLab рендерят в картинку прямо в README — без дополнительных инструментов.

> **Для тех, кто уже в теме:** Diagrams-as-code — часть более широкой идеи docs-as-code: документация хранится в репозитории, проходит code review, версионируется. Если ваша команда уже работает с документацией в Markdown и git, добавить Mermaid-диаграммы — логичный следующий шаг. Подробнее об инструментах документирования — в главе 16.

## Как не дать схемам устареть

Три простых правила:

**Правило места.** Храните схему рядом с тем, что она описывает. Если схема описывает API — в репозитории рядом с API. Если бизнес-процесс — в Confluence в той же статье, что и требования. Схема, лежащая отдельно от контекста, рано или поздно теряется.

**Правило задачи.** Если задача меняет процесс — схема меняется вместе с ней, а не «потом». «Потом» не наступает. Включите обновление схемы в Definition of Done своих задач.

**Правило доступности.** Не создавайте схемы, которые никто не откроет. Диаграмма, нарисованная в Visio и сохранённая как `.vsd` на чьём-то ноутбуке с лицензией — не артефакт, а документ-призрак. Если нет доступа — нет ценности.

---

Инструмент — средство, не цель. Выбирайте то, что вписывается в процесс вашей команды. Mermaid-схема в README, которую читают все, — лучше, чем идеальный BPMN в папке, которую никто не открывает. На этом — и завершается восьмая глава. Вы теперь знаете, зачем рисовать схемы, что и когда использовать, и где хранить, чтобы они не исчезали. Дальше — базы данных и SQL: фундамент, без которого аналитик работает вслепую.
