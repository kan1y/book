# 11.4 REST API

Если API — это меню, то REST — самый популярный способ это меню оформить. Когда сегодня говорят «дёрни ручку», «сходи в апишку», «пришли мне ответ джейсоном» — почти всегда речь про REST. Это тот стиль, с которым аналитик сталкивается чаще всего, поэтому разберём его подробнее остальных.

**REST** (Representational State Transfer) — не протокол, а **архитектурный стиль**: набор принципов, как строить API поверх обычного протокола HTTP (того самого, на котором работает весь веб). Он более гибкий и простой, чем старый SOAP, обычно передаёт данные в JSON и стал фактическим стандартом современных интеграций.

---

## Ресурсы и HTTP-методы

В основе REST лежит идея **ресурсов** — сущностей, с которыми работаем: клиенты, заказы, платежи. Каждый ресурс живёт по своему адресу (URL): `/clients`, `/clients/42`, `/orders`.

А действия над ресурсами задаются **HTTP-методами**. И здесь всплывает старый знакомый — CRUD из [[09.4 SQL с нуля — SELECT, WHERE, ORDER BY|главы про SQL]]. REST построен на тех же четырёх операциях, только называются они иначе:

| Метод | Что делает | CRUD | Пример |
|---|---|---|---|
| **GET** | получить данные | Read | `GET /clients/42` — данные клиента 42 |
| **POST** | создать новое | Create | `POST /clients` — создать клиента |
| **PUT** | заменить целиком | Update | `PUT /clients/42` — перезаписать клиента 42 |
| **PATCH** | изменить частично | Update | `PATCH /clients/42` — поменять только статус |
| **DELETE** | удалить | Delete | `DELETE /clients/42` — удалить клиента 42 |

Красота в том, что адрес говорит «с чем работаем», а метод — «что делаем». `GET /orders/17` читается почти по-человечески: «получи заказ семнадцать».

## Из чего состоит запрос

REST-запрос — это не только адрес. У него несколько частей, и аналитику полезно понимать каждую:

- **Метод и URL** — что делаем и с чем: `POST /orders`.
- **Заголовки (headers)** — служебная информация: формат данных, авторизация (токен — об этом в [[11.9 Аутентификация и безопасность API|11.9]]).
- **Тело (body)** — сами данные, обычно JSON. Например, при создании заказа в теле летит `{ "client_id": 42, "product_id": 7 }`.

В ответ приходят **статус-код** (см. ниже) и тело ответа — тоже обычно JSON.

## Статус-коды: как API говорит «получилось» или «нет»

На каждый запрос REST отвечает трёхзначным кодом. Их не нужно зубрить, достаточно знать группы — по первой цифре:

| Группа | Смысл | Частые примеры |
|---|---|---|
| **2xx** | Успех | `200 OK`, `201 Created` (создано) |
| **3xx** | Перенаправление | `304 Not Modified` |
| **4xx** | Ошибка на стороне клиента | `400` (кривой запрос), `401` (не авторизован), `403` (нет прав), `404` (не найдено) |
| **5xx** | Ошибка на стороне сервера | `500` (внутренняя ошибка), `503` (сервис недоступен) |

Ключевое различие, которое стоит усвоить: **4xx — вы что-то сделали не так** (послали кривые данные, забыли токен), **5xx — сломалось на той стороне**. Когда интеграция не работает, первый вопрос — какой код возвращается: он сразу говорит, на чьей стороне искать проблему. Это экономит часы взаимных обвинений между командами.

## Идемпотентность: защита от дублей

Одно понятие, без которого разговор про REST неполон, — **идемпотентность**. Запрос идемпотентен, если повторить его дважды безопасно: результат тот же, что и от одного вызова. Звучит академично, а на деле — защита от двойного списания денег.

- `GET`, `PUT`, `DELETE` — идемпотентны. Удалить клиента 42 один раз или три раза подряд — итог один: клиента нет.
- `POST` — **не** идемпотентен. Два `POST /orders` создадут **два** заказа.

Почему это важно аналитику: в интеграциях запросы теряются и повторяются постоянно (сеть моргнула — клиент шлёт ещё раз). Если операция «провести платёж» не защищена от повтора, клиент рискует заплатить дважды. К тому, как это закладывают в требования, вернёмся в [[11.12 Асинхронность в требованиях|11.12]] — но само слово запомните уже здесь.

## Чем это трогают руками

Аналитику необязательно писать код, чтобы работать с REST. Есть инструменты, чтобы «постучаться» в API самому:

- **Postman** — самый популярный: собираешь запрос мышкой, жмёшь Send, смотришь ответ. Незаменим, чтобы проверить, как ведёт себя ручка, до всякой разработки.
- **Swagger UI** — интерактивная документация, где методы можно и посмотреть, и тут же вызвать ([[11.8 Контракт API — OpenAPI и Swagger|11.8]]).

> **Для тех, кто уже в теме:** у REST есть нюансы, на которых легко споткнуться и о которых любят спрашивать на собеседованиях. **Пагинация** — большие списки не отдают целиком, а порциями (`GET /orders?page=2&limit=50`). **Именование ресурсов** — по соглашению существительные во множественном числе (`/orders`, не `/getOrders`); Джей Гивакс в «Паттернах проектирования API» посвящает этому целые главы, потому что кривое именование — источник вечной путаницы. **Вебхуки** (webhooks) — «обратный REST»: не вы спрашиваете сервис, а он сам стучится к вам, когда что-то произошло (платёж прошёл — платёжный сервис дёрнул ваш URL). Вебхуки — частый способ получить асинхронный ответ, не опрашивая сервис постоянно.

---

Итог. REST — это API поверх HTTP, где ресурсы адресуются через URL, а действия задаются методами GET/POST/PUT/PATCH/DELETE (те же CRUD). Ответ всегда несёт статус-код: 4xx — вопросы к клиенту, 5xx — к серверу. Держите в голове идемпотентность как защиту от дублей и пробуйте ручки руками в Postman. Дальше — про строгого предка REST, который до сих пор жив в банках: SOAP.
