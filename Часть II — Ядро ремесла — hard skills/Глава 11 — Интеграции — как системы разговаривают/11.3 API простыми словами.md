# 11.3 API простыми словами

Спросите на собеседовании, что такое API, — и половина кандидатов бодро отчеканит расшифровку: Application Programming Interface, программный интерфейс приложения. Формально верно и совершенно бесполезно для понимания. Поэтому дам определение попроще: **API — это способ сделать так, чтобы одно приложение понимало другое**. Точка. Всё остальное — детали.

---

## Метафора меню в ресторане

Лучшая аналогия для API — меню в ресторане.

Вы приходите в ресторан и хотите есть. Вы же не идёте на кухню и не объясняете повару, как включать плиту и где лежат продукты. Вы берёте **меню** — список того, что можно заказать, — и говорите официанту: «борщ, пожалуйста». Официант уносит заказ на кухню и через время приносит борщ. Как именно его готовили, вы не знаете и знать не хотите — вам важен результат.

API работает точно так же:

- **Меню** — это список операций, которые система готова выполнить («узнать баланс», «создать заказ», «отправить платёж»).
- **Ваш заказ** — это **запрос** (request) к API.
- **Кухня** — это внутренняя логика чужой системы, скрытая от вас.
- **Принесённое блюдо** — это **ответ** (response).

Главная прелесть в том, что вам не нужно знать, как чужая система устроена внутри. Достаточно знать меню — какие запросы она принимает и что возвращает. Это и есть смысл слова «интерфейс»: витрина, за которой спрятана вся сложность.

---

## Запрос и ответ

Любое обращение к API — это пара «запрос → ответ». Вы формулируете запрос по правилам, которые задала система: что вы хотите, к какому «ресурсу» обращаетесь, какие данные передаёте. Система обрабатывает и возвращает ответ: результат или сообщение об ошибке.

Простой пример словами. Приложение хочет узнать данные клиента с номером 42. Оно шлёт запрос: «дай мне клиента 42». API отвечает: «вот он: имя Иван, статус активен». Приложение получило данные и показало их на экране — при этом оно понятия не имеет, из какой базы и как их достали на той стороне.

---

## Контракт: договор между системами

Чтобы две системы понимали друг друга, они должны заранее договориться о правилах: какие запросы бывают, какие поля обязательны, что придёт в ответе, как выглядит ошибка. Этот свод правил называется **контрактом API** (о том, как его формализуют, — в [[11.8 Контракт API — OpenAPI и Swagger|11.8]]).

Контракт — ключевое понятие для аналитика. По сути, договориться об интеграции = договориться о контракте. Если стороны понимают меню по-разному (одна шлёт дату как `2024-01-05`, другая ждёт `05.01.2024`) — интеграция ломается, даже если обе системы «работают правильно» каждая по-своему.

---

## Какие бывают API

API — это идея, а реализовать её можно по-разному. Есть несколько распространённых стилей, и почти все они всплывут в вашей работе:

- **REST** — самый современный и распространённый стиль, гибкий, обычно поверх JSON ([[11.4 REST API|11.4]]).
- **SOAP** — строгий «дедушкин» протокол на XML, до сих пор живой в банках ([[11.5 SOAP и где он жив|11.5]]).
- **GraphQL** — гибкий подход, где клиент сам запрашивает ровно те данные, что нужны ([[11.6 GraphQL коротко|11.6]]).

Их и разберём по очереди в следующих подглавах. Пока держите в голове главное: под всеми этими словами прячется одна и та же идея меню-запроса-ответа, просто оформленная по-разному.

---

Итог. API — это меню, через которое одно приложение заказывает что-то у другого, не заглядывая на кухню. Обращение к нему — всегда пара «запрос → ответ», а работает всё это благодаря контракту: заранее согласованным правилам обмена. Разобравшись с идеей, переходим к её самой ходовой реализации — REST.
