# 11.7 Форматы данных: JSON и XML

Мы уже несколько раз помянули JSON и XML — пора разобрать их отдельно. Это два самых распространённых формата, в которых системы передают данные друг другу. Аналитик читает их каждый день: в ответах API, в логах, в примерах к контрактам. Уметь читать оба и не пугаться — базовый навык.

Оба решают одну задачу — записать структурированные данные так, чтобы их поняла и машина, и (при желании) человек. Просто делают это по-разному.

---

## JSON: лёгкий и современный

**JSON** (JavaScript Object Notation) — сегодня формат по умолчанию, особенно в REST. Он устроен на двух простых идеях: **пары «ключ — значение»** и **вложенность**.

```json
{
  "id": 42,
  "name": "Иван Петров",
  "active": true,
  "balance": 1500.50,
  "orders": [
    { "order_id": 501, "sum": 1200 },
    { "order_id": 502, "sum": 340 }
  ]
}
```

Читается почти как обычный текст: у клиента есть `id`, `name`, флаг `active`, баланс и список заказов. Несколько правил, которые стоит знать:

- Данные — в фигурных скобках `{}`, списки — в квадратных `[]`.
- Ключ всегда в кавычках, значение — по типу: текст в кавычках, числа и `true/false` без.
- Структуры вкладываются друг в друга (заказы внутри клиента).

За счёт минимума лишних символов JSON компактен, быстро читается и людьми, и машинами. Поэтому его и любят.

---

## XML: строгий и многословный

**XML** (eXtensible Markup Language) — формат постарше, где данные размечаются **тегами** (как в HTML): у каждого значения есть открывающий и закрывающий тег. Тот же клиент в XML:

```xml
<client>
  <id>42</id>
  <name>Иван Петров</name>
  <active>true</active>
  <balance>1500.50</balance>
  <orders>
    <order>
      <order_id>501</order_id>
      <sum>1200</sum>
    </order>
  </orders>
</client>
```

Те же данные — но заметно многословнее: каждое поле обёрнуто в пару тегов. Зато XML умеет то, чего у JSON нет из коробки: **строгую проверку по схеме** (XSD) — можно формально задать, какие теги обязательны и какого они типа. Именно за эту строгость XML держат в SOAP и банковских стандартах вроде ISO 20022 ([[11.5 SOAP и где он жив|11.5]]).

---

## JSON или XML: как выбирают

| | JSON | XML |
|---|---|---|
| Многословность | Низкая (компактный) | Высокая (теги) |
| Где чаще | REST, веб, мобилки | SOAP, банки, легаси |
| Проверка по схеме | Опционально (JSON Schema) | Сильная сторона (XSD) |
| Читаемость человеком | Очень высокая | Средняя |

Правило простое: современный REST-сервис — почти наверняка JSON; SOAP и «взрослые» банковские интеграции — XML. Выбор обычно диктует не вкус, а то, к чему подключаешься.

> **Для тех, кто уже в теме:** аналитику полезно уметь не только читать, но и **валидировать** эти форматы. Сломанная запятая в JSON или незакрытый тег в XML роняют разбор целиком — а онлайн-валидатор или тот же Postman ловят это за секунду. Ещё встретите форматы-родственники: **YAML** (на нём пишут спецификации OpenAPI и конфиги — по сути тот же JSON, но человеко-дружелюбнее) и упомянутый в контексте gRPC бинарный **Protobuf** (компактный, но нечитаемый глазом). Разбираться в них глубоко не нужно — достаточно узнавать в лицо.

---

Итог. JSON — компактные пары «ключ — значение», формат современного REST; XML — многословная разметка тегами со строгой проверкой по схеме, вотчина SOAP и банков. Аналитику нужно свободно читать оба и уметь быстро проверить их на валидность. Мы разобрали, *как* системы разговаривают и *в чём* передают данные, — пора формализовать сами договорённости: контракт API.
